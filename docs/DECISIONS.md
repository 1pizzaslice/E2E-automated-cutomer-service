# Architecture Decision Log

## Purpose

This file records decisions that future agents should not re-litigate unless new evidence or user direction changes the context. Add a new entry whenever architecture defaults, public contracts, platform choices, or operating rules change.

## Decision Format

Use this format:

```md

- Date:
- Status: Accepted | Superseded | Proposed
- Context:
- Decision:
- Consequences:
- Follow-up:
```

- Date: 2026-06-18

## ADR-0013: Add PostgreSQL Row-Level Security Before Tenant APIs

- Date: 2026-06-19
- Status: Accepted
- Context: Repository helpers now enforce tenant filters and live PostgreSQL tests prove those helpers do not return cross-tenant rows. The platform will still expose high-risk multi-tenant support data through API endpoints, workflows, AI retrieval, tool execution, and audit reads.
- Decision: Add PostgreSQL row-level security policies for tenant-scoped tables before exposing tenant-scoped API endpoints. Repository tenant filters remain mandatory; RLS is a database-level defense in depth, not a replacement for scoped data access.

- Follow-up: Milestone 3 API and worker database code must set transaction-local tenant context before tenant-scoped operations.

## ADR-0014: KB Ingestion Uses A Deterministic Embedder Port, Content By Reference, And An HNSW Cosine Index

- Date: 2026-07-03
- Status: Accepted
- Context: Milestone 7 KB ingestion must chunk and embed tenant documents into `kb_chunks` for retrieval. A hosted embedding model is a network dependency that is non-deterministic across versions and unavailable in unit/CI runs, so ingestion built directly on it would be unreproducible and untestable. Raw document bodies can be large and do not belong inline in PostgreSQL. Retrieval needs an index for approximate nearest-neighbour search.
- Decision: (1) Chunking and embedding are pure, deterministic functions in `@support/integrations/kb` behind an `Embedder` port; the v1 default is `createDeterministicEmbedder` (a token-hash bag-of-words unit-vector embedder producing `vector(1536)`), and production swaps a hosted model behind the same interface. (2) Raw KB document content is stored by reference in a `KbContentStore` port (filesystem default, keyed by tenant + document id), never inline in the DB; the row keeps only metadata and a `content_hash`. (3) Retrieval is indexed with a pgvector HNSW index using `vector_cosine_ops` (`0003_kb_vector_index.sql`), because the v1 embedder emits L2-normalized vectors so cosine is the correct metric.
- Consequences: Ingestion is reproducible and unit-testable without a model or network, and remains replay-safe when a Temporal `KbIngestionWorkflow` later drives the same steps as activities. Similarity scores from the deterministic embedder are lexical, not semantic, so retrieval quality on real client data depends on choosing and documenting a production embedding model before pilot; the `vector(1536)` dimension and cosine metric must match whatever model is chosen. Content-store durability and access control are separate concerns from the DB.
- Follow-up: Retrieval half delivered in ADR-0015. Choose and document a production embedding model; re-embed if its dimension differs from 1536.

## ADR-0015: KB Retrieval Ranks By Cosine, Excludes Stale Documents At Query Time, And Treats Chunk Content As Untrusted

- Date: 2026-07-03
- Status: Accepted
- Context: Milestone 7's retrieval half must let the AI runtime find evidence in `kb_chunks` without ever crossing a tenant boundary, without surfacing retired knowledge, and without letting adversarial document text ("prompt injection") change system behavior. The v1 `Embedder` emits L2-normalized `vector(1536)` embeddings indexed by an HNSW `vector_cosine_ops` index (ADR-0014). A KB document's lifecycle `status` is mutated by PATCH on the `kb_documents` row only; a re-ingest replaces chunk rows but a plain status change to `stale`/`archived` does not touch existing chunk rows.
- Decision: (1) Retrieval is a tenant-scoped cosine (`<=>`) nearest-neighbour query (`searchKbChunksQuery`) exposed as `POST /v1/kb/search` behind a new `kb:search` permission; it embeds the query with the same `Embedder` used at ingestion and runs under `withTenantTransaction`/RLS with explicit `tenant_id` predicates as defense in depth. (2) Stale/inactive documents are excluded at query time by inner-joining `kb_documents` and requiring both `kb_chunks.status = 'active'` and `kb_documents.status = 'active'` — so a document PATCHed to `stale` drops out of answers even though its chunk rows remain, and no chunk rewrite is needed. (3) Results are `KbChunkResponseSchema`-based with a relevance `score` (cosine similarity = `1 - distance`) plus document-level citation fields (`document_title`, `document_type`, `source_type`, `source_ref`) so answers are attributable. (4) Retrieval treats chunk content as untrusted data: it returns adversarial text verbatim as evidence and never interprets it; ranking is relevance-only, so injected documents cannot hijack results. Downstream guardrails (the Milestone 9 AI runtime) are responsible for refusing to obey retrieved instructions.
- Consequences: Retrieval is reproducible and network-free in tests via the deterministic embedder; a golden eval corpus + labeled queries and adversarial prompt-injection fixtures guard ranking and injection-resilience, and tenant isolation is proven by unit and live-pgvector tests. Because stale exclusion is a query-time join, retiring a document is instant and reversible via PATCH with no re-embedding. The deterministic embedder is lexical, so real retrieval quality still depends on choosing a production embedding model (ADR-0014 follow-up) wired into both ingestion and retrieval behind the `Embedder` port. Raw embeddings are never returned to API clients or the AI runtime.
- Follow-up: A Temporal `KbIngestionWorkflow`; wire one shared production embedder instance into both ingestion and retrieval; consider a similarity-score floor and a max-context-tokens cap before retrieved chunks enter the AI runtime.

## ADR-0016: V1 AI Runtime Is A Dependency-Free LangGraph-Style Graph Behind Pluggable Ports

- Date: 2026-07-03
- Status: Accepted
- Context: Milestone 9 implements the Python AI runtime (the support agent graph). The default stack names LangGraph for stateful agent orchestration (ADR-0005) and Pydantic-style validation. However, the local engineering harness runs Python through the system interpreter (`pnpm test:py` → `python3 -m unittest`, `pnpm lint` → `python3 -m compileall ai`) on Python 3.14 with no `uv` and no installed third-party packages; LangGraph, LangChain, and Pydantic are therefore not importable, and adding them would make `pnpm test`/`pnpm lint` fail and be non-reproducible for future fresh-context sessions. The runtime's value (state model, node governance, structured outputs, guardrails, traces, evals, escalation) is independent of the specific orchestration library, and the harness explicitly deferred Python dependency management to "when the LangGraph runtime is implemented" (AGENTS.md / TEST_STRATEGY.md).
- Decision: Implement the v1 support graph as a self-contained, dependency-free Python package under `ai/runtime/` that mirrors LangGraph's node model. (1) A tiny graph engine (`graph.py`) reproduces the LangGraph API surface the graph needs — `add_node`/`set_entry_point`/`add_edge`/`add_conditional_edges`/`compile().invoke()` — so nodes are written exactly as they would be for LangGraph. (2) Structured I/O is validated with stdlib dataclasses + explicit validators (Pydantic-equivalent) in `schemas.py`, and `RuntimeResult` mirrors the Temporal `RunAiGraphActivityResult` boundary. (3) The three external dependencies are ports: `ModelProvider` (classification + drafting), `RetrievalPort` (tenant-scoped KB search), and `ToolExecutor` (the Milestone 8 tool-call envelope + permission classes). V1 ships deterministic, offline implementations (`DeterministicSupportModel`, `InMemoryRetrieval`, `InMemoryToolExecutor`) so the graph, evals, and tests run with no network, no wall-clock, and no randomness. (4) Model reasoning goes through the model port; policy and guardrail logic is deterministic Python (safety-critical governance must not be probabilistic). (5) Traces and ids are deterministic (derived from inputs) so runs are reproducible and diffable.
- Consequences: `pnpm test`/`pnpm lint` stay green with the system `python3`, and the runtime is fully unit-/eval-testable offline (49 tests + a golden dataset with hard-fail safety gates). The ports are the exact seams where production adapters plug in: a real LLM-backed `ModelProvider`, a `RetrievalPort` calling `POST /v1/kb/search`, and a `ToolExecutor` calling the live TypeScript tool registry — and, if desired, the real LangGraph library can replace the local engine without changing node code. This partially supersedes the _implementation mechanism_ of ADR-0005 (real LangGraph is deferred, not rejected); ADR-0005's principle (LangGraph-style stateful graph returning structured outputs to Temporal/backend, not owning durable state) is upheld. The deterministic model is rule-based, so real classification/draft quality still depends on wiring a production model behind the port.
- Follow-up: Python dependency management is now provisioned — `uv` is installed (`~/.local/bin`), `ai/.python-version` pins uv-managed CPython 3.12, `ai/uv.lock` is committed, and the harness runs the suite via `uv run --frozen --project ai`; the real AI stack (pydantic, langgraph, langchain-core) is proven to install and is available with `uv sync --project ai --extra llm`. This unblocks (but does not by itself perform) swapping real LangGraph + a provider SDK behind the ports. Remaining: swap the real model/graph adapters; call the live TS tool registry and `/v1/kb/search`; wire the runtime behind the Temporal `RunAiGraphActivity` placeholder; expand the golden dataset to the TEST_STRATEGY §4 counts and add an LLM-graded draft rubric; add versioned prompt files (harness §8).

## ADR-0017: Approval Decisions Persist First, Then Signal; Outbound Sends Reuse The Messages Table With Deterministic Idempotency

- Date: 2026-07-04
- Status: Accepted
- Context: Milestone 10 closes the human approval loop: reviewers resolve pending approvals through the API, the waiting `ticketLifecycleWorkflow` must resume, and approved/edited responses must be delivered to the customer exactly once over email/WhatsApp with a full audit trail. This spans two failure-prone boundaries — the API→Temporal signal (a classic dual-write) and the provider send inside a retryable Temporal activity — and three consumers of the same truth (API, workflow activities, QA/eval).
- Decision: (1) Approval decisions are **persist-first**: one tenant/RLS transaction resolves the approval (guarded by `status = 'pending'` in the UPDATE so concurrent double-decides conflict deterministically instead of overwriting a reviewer) and appends the `approval.{status}` audit event whose metadata carries both `requested_payload` and `approved_payload` — the edited-draft trail QA/eval consumes; only after commit does the API signal `approval_completed` (workflow id `ticket-lifecycle:{tenant}:{conversation}`) through an injectable `ApprovalWorkflowSignaler`. A missing workflow is a reported outcome (`workflow_signal.reason = "workflow_not_found"`), not an error, because seeded/manual approvals have no waiting workflow; transport failures surface as `502 WORKFLOW_ERROR` after the decision is already durable. (2) Outbound messages are rows in the existing `messages` table (`direction: "outbound"`), not a new table: the pre-existing partial unique index on `(tenant_id, idempotency_key)` is the send-once guarantee for the workflow's deterministic `outbound:{tenant}:{ticket}:{approval_id}` key, and the row carries `send_status` (`queued → sent|failed`), `provider_message_id`, `approval_id`, and `ai_run_id` linkage. (3) The production activity implementations (`createApproval`, `sendOutboundMessage`, `recordAuditEvent`) live behind a `TicketLifecyclePersistenceStore` port (database + in-memory) and derive **deterministic ids** — approval id from (tenant, ticket, correlation), audit ids from a stable hash of the write — so Temporal at-least-once retries replay instead of duplicating; a `sent` idempotency key replays the recorded outcome without re-contacting the provider, while a `failed` row is reused by the next retry. (4) Channel delivery goes through pure provider-request builders plus an `OutboundChannelSender` port; the HTTP sender (Mailgun, WhatsApp Cloud) takes an injectable `fetch`, classifies failures as retryable (5xx/transport) vs permanent (4xx/config/credential → `NonRetryableActivityError`), and resolves credentials from the channel config's `send_credential_ref` via an env-backed resolver (no plaintext secrets in rows, §4.1). (5) `approvals.ai_run_id`/`messages.ai_run_id` are linked only when the referenced `ai_runs` row exists (it does not until AI-run persistence lands); the id always remains inside the approval payload, so nothing is lost.
- Consequences: The approval record is the source of truth and every decision is auditable even if signaling fails; the worst dual-write failure mode is a resolved approval whose workflow needs signal redelivery (surfaced by the 502), never a sent-but-unrecorded or double-sent message. Send-once holds across API retries, Temporal activity retries, and concurrent workflows because it is enforced by a database unique index rather than in-process state. The in-memory store and recording sender/signaler keep the whole loop offline-testable; the live workflow suite plus a live-PostgreSQL drive verified pause→resume, send-once, no-send-on-reject, and audit dedup end to end. Trade-offs accepted: audit rows for retried-but-different metadata hash to distinct ids (at-least-once, append-only is fine); outbound email subjects are null until a subject strategy lands (threading relies on RFC 5322 reply headers); `send_status`/`sent_by_type` remain free-text columns pending an enum migration.

## ADR-0018: Observability Is A Shared Port Package; Cross-Process Traces Correlate By Attribute; QA Sampling Is Deterministic

- Date: 2026-07-04
- Status: Accepted
- Context: Milestone 11 adds OpenTelemetry tracing, structured logs with trace ids, metrics for APIs/workflows/AI runs/tool calls/approvals, AI-run persistence with trace links, a QA review surface, and dashboards/alerts. The Compose stack already ships an OTel collector (OTLP 4317/4318) but no Prometheus/Grafana. The repo's testing doctrine requires all of this to be deterministic and offline-testable, and Temporal's workflow sandbox makes strict parent-child span propagation across the API→workflow boundary a nontrivial interceptor integration.
- Decision: (1) A new `@support/observability` package owns the telemetry seam: `startTelemetry` (OTLP/HTTP exporters to the collector, registered global providers, disabled via `OTEL_SDK_DISABLED`), `createInMemoryTelemetry` (test double capturing spans/metrics), a typed `SupportMetrics` port (OTel-backed, no-op, and recording implementations) with canonical instrument names in `SUPPORT_METRIC_NAMES` (`support.{domain}.{measure}`), a JSON structured logger (required fields per DEVELOPMENT_RULES §13, secret-key redaction, active-span trace-id injection), and shared `SUPPORT_ATTR` span attribute keys. Instrumented code takes the port with a no-op default, so telemetry is never required for correctness. (2) V1 tracing correlates across process boundaries **by attribute, not by span parentage**: every span (API `http.request`, worker `activity.*`, `tool.execute`, `approval.decide`) carries `support.correlation_id`/`support.tenant_id`/`support.ticket_id`, `ai_runs.trace_id` joins the persisted run to its spans and to the runtime's redacted trace export, and `audit_events.correlation_id` closes the loop — a ticket is traced end to end by filtering on one correlation id. In-process spans still nest via the active context. Strict W3C context propagation through Temporal (`@temporalio/interceptors-opentelemetry`) is deferred as a follow-up, not rejected. (3) Workflow observability instruments the **activity boundary**: `instrumentTicketLifecycleActivities` wraps every activity with spans/duration/outcome metrics and maps domain outcomes to a closed set of critical failure modes (`ai_graph_failed`, `outbound_send_failed`, `approval_signal_failed`, `event_dead_letter`, `sla_breached`) recorded on one `support.critical_failures` counter that the alert rules key on. Workflow code itself stays deterministic and untouched. (4) `ai_runs` persistence wraps `runAiGraph` (`createPersistedRunAiGraph`): deterministic run ids make Temporal retries replay, failed runs are persisted as evidence with a backfilled deterministic id, and a missing ticket row skips persistence instead of failing the workflow — this is what materializes the Milestone 10 `approvals.ai_run_id`/`messages.ai_run_id` FK links. (5) QA sampling is a deterministic job, not a scheduler: mandatory rules (100% auto-send candidates, 100% high-risk) plus a hash-bucketed random sample (SOP §10 rate), deterministic `qa_review_id` + conflict-safe inserts make re-runs idempotent, and each new review emits `support.qa.review_created.v1`. (6) Dashboards and alert rules are definitions-as-code under `infra/observability/`; the collector re-exposes metrics for Prometheus scraping on `:8889` with `translation_strategy: UnderscoreEscapingWithoutSuffixes` so scraped names match the code's instrument names dot-for-underscore.
- Consequences: Every instrumented path works with telemetry disabled and is asserted offline via the recording/in-memory doubles; the live smoke drive proved spans and metrics arrive at the collector and scrape under the documented names. Flat cross-process traces mean no single waterfall view across API→worker until the Temporal interceptor lands — the trade is zero risk to workflow determinism/replay and no new workflow-bundle dependencies. Metric cardinality stays bounded (route templates, enum attributes). The QA surface (`qa_reviews` CRUD + composite evidence endpoint) gives reviewers conversation, evidence, tool calls, AI output, human edits, and the final response in one tenant-scoped read.
- Follow-up: Adopt `@temporalio/interceptors-opentelemetry` for strict span parentage across the Temporal boundary; ship Prometheus/Grafana (or a hosted backend) in pilot infra and load `infra/observability/`; emit `support.ai_run.completed.v1`/`support.tool_call.completed.v1` domain events from the workflow; wire the workers' structured logger into a production worker entrypoint when it lands (Milestone 10 follow-up); add log export (OTLP logs) once a log backend exists.

## ADR-0019: Security Controls Fail Closed — Deny-By-Default RBAC, Closed Audit Taxonomy, Policy-Version Auto-Send Allowlist, and Safe-Subset Retention

- Date: 2026-07-04
- Status: Accepted
- Context: Milestone 12 hardens the platform for the first pilot: RBAC verification, integration secret handling, PII redaction, a prompt-injection suite, attachment validation, audit completeness, data retention hooks, pilot seed data, the weekly review report, auto-send allowlist controls, and a deployment checklist. The acceptance criteria are absolute ("no known cross-tenant leakage", "no high-risk action can bypass human approval"), so every new control has to default to the safe side and be mechanically checkable, not just documented.
- Decision: (1) **RBAC is deny-by-default with a self-verifying matrix.** The implicit `support_agent` fallback for a missing/empty `x-user-roles` header is removed — no parseable role is `401`, so a misconfigured gateway cannot mint access. `ROLE_PERMISSIONS` is exported as the single source of truth, a new `reports:read` permission gates reporting, and an RBAC matrix test enumerates every registered route via a Fastify `onRoute` collector against a permission catalog: an uncataloged route, a role passing without its permission, or a role denied with it all fail the suite, so adding an endpoint forces an explicit permission decision. (2) **Audit actions are a closed taxonomy.** `SupportAuditActionSchema` in shared-schemas is the canonical action set; the workers audit boundary is typed to it at compile time, the API decide path validates against it at runtime, reserved entries (`policy.*`, `integration.credential_changed`, `permission.*`) pre-commit the families whose write paths are pending, and an audit-completeness test drives every live producer (tool calls stay audited in `tool_calls`, Milestone 8). (3) **Auto-send controls live in the policy model and fail closed.** The tenant allowlist is the active `automation`-domain `policy_versions.content` validated by `AutomationPolicyContentSchema` — an `auto_send_enabled` kill switch plus topics constrained at the contract level to the closed low-risk set (`faq | order_status`), mirroring the AI runtime's hardcoded ceiling so a tenant cannot allowlist `refund` even deliberately (proven by golden case `auto_2`). `GET /v1/policies/automation` resolves the effective controls (missing/inactive/malformed → disabled); `evaluateAutoSendEligibility` in workers is the single gate a future auto-send branch must consult (kill switch → succeeded run → explicit `auto_send` recommendation → low risk → guardrails passed → draft present → allowlisted topic). The v1 workflow still creates an approval unconditionally — a live workflow test drives an `auto_send` recommendation and proves send is unreachable without a human signal. (4) **Retention executes only the reversible-safe subset.** `tenants.retention_policy` (jsonb, migration 0004, `TenantRetentionPolicySchema`) configures per-class windows; the job clears expired `messages.raw_payload_ref` in bounded batches (returning refs for the blob sweeper) and audits `retention.applied`, while attachment/AI-run purges are counted as placeholders until blob-deletion/anonymization strategies exist. No or malformed configuration purges nothing. (5) **Secrets are validated references; PII redaction is content-level.** Both existing resolvers delegate to one `SecretResolver` (`packages/integrations`) that requires `^[A-Z][A-Z0-9_]*$` before touching the environment; the structured logger scrubs emails/phones/card-like digit runs from string content (and the message itself) on top of non-disableable secret-key redaction. (6) **Intake validates attachments before persistence.** A pure `validateInboundAttachments` policy (size cap, content-type allowlist, filename safety, per-message bound) runs at the top of `ingestNormalizedMessage`, so a rejected message creates nothing and is reported per message in the webhook `202` (`rejected`/`rejection_reason`) instead of a retry-provoking error status. (7) **Pilot readiness is executable.** An idempotent, deterministic seed plan (`buildPilotSeedPlan`/`applyPilotSeed`, `pnpm db:seed:pilot`) provisions the pilot tenant end to end — including the global first-party `tool_definitions` rows (closing a Milestone 8 follow-up) and the disabled automation policy — and the weekly pilot report is a tenant-scoped aggregate service behind `GET /v1/reports/pilot-weekly` computing the SOPS §14 metrics in one RLS transaction.
- Consequences: Every Milestone 12 control defaults to deny/disabled/no-purge and has negative tests; the acceptance criteria are backed by suites rather than review (cross-tenant: RLS + RBAC matrix + isolation integration tests; no-bypass: workflow no-bypass test + eligibility gate + Python hard-fail gates incl. the 18-case injection suite; reportability: report endpoint tests; SOPs: SOPS §1.1/§13/§19). Trade-offs accepted: requests that previously relied on the implicit default role now 401 (deliberate breaking change at a placeholder auth boundary — real token verification remains future work at the same choke point); PII scrubbing can redact rare non-PII 13+ digit runs (epoch millis) — an accepted false-positive cost; retention leaves attachment/AI-run data in place until purge strategies land; allowlist changes require an ops-applied policy version until policy write/approve/activate endpoints exist.
- Follow-up: real bearer-token verification and tenant-membership checks in `readAuthContext`; policy write/approve/activate endpoints emitting the reserved `policy.*` audit actions; the auto-send workflow branch behind `evaluateAutoSendEligibility` with shadow mode first (SOPS §17 ladder); blob deletion + anonymization to execute the planned retention classes; scheduling the retention job alongside QA sampling; surfacing `retention_policy` on the tenant API contract.

## ADR-0020: V1 Launch Plan — Sidecar AI Bridge, Provider-Agnostic Model Layer, Hosted-IdP Auth, Single-VM Pilot Deployment

- Date: 2026-07-04
- Status: Accepted
- Context: All twelve build milestones are complete: every subsystem is implemented and verified, but the system has never run as one live process end to end. The remaining gaps between "all parts verified" and "a customer email becomes an approved, sent reply in production" are integration and operations: no worker entrypoint composes the activities, ticket state is not persisted from the workflow, the AI runtime has only deterministic offline providers, auth is a trusted-header placeholder, jobs have no scheduler, and no deployed environment exists. The reviewer console will be built in a separate repository, keeping this repo backend-only (ADR-0001). Four platform choices were needed to plan the launch milestones; each was decided with the user on 2026-07-04.
- Decision: Launch engineering is organized into four phases tracked as `TODO.md` Milestones 13-22 — Phase 1 "Run End-To-End" (13: production worker entrypoint + ticket persistence; 14: AI runtime service bridge; 15: provider-agnostic model layer + real LLM default; 16: real authentication + policy lifecycle; 17: scheduled jobs + retention execution), Phase 2 "Deploy" (18: staging on hardened Compose; 19: live providers + go-live rehearsal), Phase 3 "Console Enablement" (20: console-enablement API), Phase 4 "Pilot" (21: eval expansion + shadow replay; 22: pilot gap-closing + go-live) — with these platform decisions: (1) **The AI bridge is an HTTP sidecar.** The Python runtime runs as a FastAPI service (`POST /internal/ai/run`, internal bearer token); the TypeScript `runAiGraph` activity calls it over HTTP. Chosen over a Python Temporal worker because the Milestone 10/11 activity plumbing (`createPersistedRunAiGraph`, `instrumentTicketLifecycleActivities`, AI-run persistence and metrics) stays unchanged, and the service is directly reusable by the eval runner and shadow replay. (2) **Model and embedding providers are config-driven behind the existing ports.** The Python `ModelProvider` is implemented over LangChain's `init_chat_model` with provider/model from env; the TypeScript `Embedder` is an env-selected factory restricted to 1536-dim-capable models (the `vector(1536)` column is the deployment standard). Pilot defaults: Anthropic Claude (LLM) and OpenAI `text-embedding-3-small` (embeddings). Provider swaps are config changes gated by a mandatory golden + injection eval re-run (SOPS §11); embedding swaps additionally require a full re-embed, so the embedding model id is recorded at ingestion and ingestion/retrieval are enforced to match. The deterministic offline implementations remain the test/CI defaults. (3) **Auth is hosted-IdP JWT verification via JWKS** (default Clerk): `readAuthContext` verifies issuer/audience/expiry against the IdP's JWKS and maps the token subject to the existing `users`/role rows (the DB stays the role source of truth); tenant membership is enforced server-side; internal service-to-service endpoints use machine tokens distinct from user tokens; the header-injection mode survives only behind an explicit env flag for tests. No passwords are stored in this platform, and the console repo gets its login story from the same IdP. (4) **Pilot deployment is a single VM running a hardened Compose profile**: production Dockerfiles, TLS reverse proxy, internal-only networks, Prometheus/Grafana/Alertmanager loading `infra/observability/`, secrets as env files per the SecretResolver contract, nightly offsite PostgreSQL backups with a tested restore, and CI image deploys with tested rollback. Chosen over PaaS/managed services for pilot economics and because the Compose stack already defines every service; managed PostgreSQL or Temporal Cloud remain drop-in upgrades if the pilot demands them.
- Consequences: Milestones 13-22 have unambiguous scope, and every accumulated milestone follow-up now lives in a specific checklist. The sidecar adds one HTTP hop and one deployable unit; its failure mode is a structured failed AI run routed to human, never a workflow failure. Provider agnosticism is plumbing-level only — prompt quality and eval thresholds are per-model, so the SOPS §11 eval gate is what makes swaps safe. The single-VM choice accepts a manual-failover recovery story (mitigated by the backup/restore drill) in exchange for pilot cost. User-owned tracks (provider/IdP accounts, VM + DNS, pilot client signing, KB collection, the console repository) run in parallel and are tracked in `TODO.md`'s launch plan section; the long-lead items (Meta Business verification, pilot client signing) start immediately.
- Follow-up: Each milestone session records its own implementation ADRs where defaults are refined (exact IdP vendor at Milestone 16; LangGraph engine swap-or-defer at Milestone 15; single-vs-multi ticket per conversation at Milestone 22). Auto-send remains out of v1 scope; the SOPS §17 ladder starts only after pilot QA data supports it.

## ADR-0021: Workflow-Owned Ticket State Persists Through An Explicit Transition Activity; Approval Expiry Is History-Recorded Config; The M13 AI Graph Is A Deterministic TypeScript Stand-In

- Date: 2026-07-04
- Status: Accepted
- Context: Milestone 13 turns the verified workflow pieces into one runnable worker process with real ticket persistence. Three design questions needed answers: (1) how workflow-owned status transitions reach the `tickets`/`ticket_events` rows (BACKEND_SPEC §6.2 requires a ticket event + audit event per transition, and the workflow previously persisted nothing); (2) how approval expiry (BACKEND_SPEC §12) gets a configurable wait inside a deterministic workflow; (3) what implementation backs `runAiGraph` in the production entrypoint, given the Python sidecar bridge is Milestone 14 and the acceptance criterion is "one process runs the entire lifecycle with the deterministic AI model".
- Decision: (1) **Transitions are an explicit `applyTicketStateTransition` activity** the workflow calls at each state change (`waiting_ai` before the AI graph, `waiting_human` at approval request, `waiting_customer` after the send, `closed` on close request; triage and customer-reply moves are folded into `runInitialTriage`/`recordInboundMessage`, and creation into `createOrUpdateTicket`). The store reads the current row status as `from_status` (the row owns truth; the workflow owns sequencing), applies the update plus the `ticket_events` row and the canonical audit event in one RLS transaction, and no-ops when the ticket is already in `to_status` — deterministic `tev_`/`aud_` ids keyed by a per-transition-point `transition_key` make Temporal retries replay instead of duplicate. The store deliberately does not enforce the §6.2 transition matrix: the workflow drives legal sequences, and recording what actually happened beats rejecting recovery paths. (2) **Approval expiry is returned by `createApproval`** (`expires_in_ms`, resolved from `APPROVAL_EXPIRY_MS`, default 24h, non-positive disables) so the value lands in workflow history and replays deterministically; the approval wait races decision signals against the SLA deadline and the expiry deadline as absolute workflow-clock timestamps. `expireApproval` reuses the pending-status guard from the API decide path, so a reviewer decision that wins the race is never overwritten — the workflow then keeps waiting for the signal that is already committed. Expired approvals audit `approval.expired` (a new closed-taxonomy action) and end the workflow in an `approval_expired` phase with the ticket left in `waiting_human`. (3) **`createDeterministicRunAiGraph` is a TypeScript stand-in** mirroring the Python deterministic model's governance (templated commitment-free drafts, always `human_approve`, no tools, medium risk for refunds/cancellations), sharing keyword rules with the new `classifyInitialTriage` triage classifier so triage and drafting agree on topics. It plugs in behind `createPersistedRunAiGraph` unchanged, which is exactly the seam the Milestone 14 sidecar replaces.
- Consequences: Every workflow-owned transition is now queryable (`tickets.status`, `ticket_events`, audit events through the existing read APIs) and idempotent under at-least-once activity execution, at the cost of one extra activity execution per transition (~4 per happy-path workflow — accepted for auditability). Approval expiry changes no default behavior in existing tests (fakes return `expires_in_ms: null`) and is disabled by setting `APPROVAL_EXPIRY_MS=0`. The deterministic TS graph means Milestone 13's live end-to-end drive proves the entire composition — webhook → persisted ticket → drafted approval → API decision → provider send → audit/event trail, across a worker restart — before any network AI dependency exists; its outputs are throwaway by design and never eval-gated.
- Follow-up: Milestone 14 swaps the inner `runAiGraph` for the HTTP sidecar; Milestone 15 replaces both the triage keyword rules and the deterministic drafts with the real model behind the same ports. Multi-run conversations (a customer reply after workflow completion starts a fresh run that re-triages the existing ticket) stay within v1's one-workflow-per-conversation model; refining reopened-ticket semantics is a Milestone 22 item per ADR-0020.
