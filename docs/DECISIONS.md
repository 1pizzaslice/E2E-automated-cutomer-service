## ADR-0013: Add PostgreSQL Row-Level Security Before Tenant APIs

- Date: 2026-06-19
- Status: Accepted
- Context: Repository helpers now enforce tenant filters and live PostgreSQL tests prove those helpers do not return cross-tenant rows. The platform will still expose high-risk multi-tenant support data through API endpoints, workflows, AI retrieval, tool execution, and audit reads.

## ADR-0014: KB Ingestion Uses A Deterministic Embedder Port, Content By Reference, And An HNSW Cosine Index

- Date: 2026-07-03
- Status: Accepted
- Context: Milestone 7 KB ingestion must chunk and embed tenant documents into `kb_chunks` for retrieval. A hosted embedding model is a network dependency that is non-deterministic across versions and unavailable in unit/CI runs, so ingestion built directly on it would be unreproducible and untestable. Raw document bodies can be large and do not belong inline in PostgreSQL. Retrieval needs an index for approximate nearest-neighbour search.
- Decision: (1) Chunking and embedding are pure, deterministic functions in `@support/integrations/kb` behind an `Embedder` port; the v1 default is `createDeterministicEmbedder` (a token-hash bag-of-words unit-vector embedder producing `vector(1536)`), and production swaps a hosted model behind the same interface. (2) Raw KB document content is stored by reference in a `KbContentStore` port (filesystem default, keyed by tenant + document id), never inline in the DB; the row keeps only metadata and a `content_hash`. (3) Retrieval is indexed with a pgvector HNSW index using `vector_cosine_ops` (`0003_kb_vector_index.sql`), because the v1 embedder emits L2-normalized vectors so cosine is the correct metric.
- Consequences: Ingestion is reproducible and unit-testable without a model or network, and remains replay-safe when a Temporal `KbIngestionWorkflow` later drives the same steps as activities. Similarity scores from the deterministic embedder are lexical, not semantic, so retrieval quality on real client data depends on choosing and documenting a production embedding model before pilot; the `vector(1536)` dimension and cosine metric must match whatever model is chosen. Content-store durability and access control are separate concerns from the DB.
- Follow-up: Retrieval half delivered in ADR-0015. Choose and document a production embedding model; re-embed if its dimension differs from 1536.

## ADR-0015: KB Retrieval Ranks By Cosine, Excludes Stale Documents At Query Time, And Treats Chunk Content As Untrusted

- Date: 2026-07-03
- Status: Accepted
- Context: Milestone 7's retrieval half must let the AI runtime find evidence in `kb_chunks` without ever crossing a tenant boundary, without surfacing retired knowledge, and without letting adversarial document text ("prompt injection") change system behavior. The v1 `Embedder` emits L2-normalized `vector(1536)` embeddings indexed by an HNSW `vector_cosine_ops` index (ADR-0014). A KB document's lifecycle `status` is mutated by PATCH on the `kb_documents` row only; a re-ingest replaces chunk rows but a plain status change to `stale`/`archived` does not touch existing chunk rows.
- Decision: (1) Retrieval is a tenant-scoped cosine (`<=>`) nearest-neighbour query (`searchKbChunksQuery`) exposed as `POST /v1/kb/search` behind a new `kb:search` permission; it embeds the query with the same `Embedder` used at ingestion and runs under `withTenantTransaction`/RLS with explicit `tenant_id` predicates as defense in depth. (2) Stale/inactive documents are excluded at query time by inner-joining `kb_documents` and requiring both `kb_chunks.status = 'active'` and `kb_documents.status = 'active'` — so a document PATCHed to `stale` drops out of answers even though its chunk rows remain, and no chunk rewrite is needed. (3) Results are `KbChunkResponseSchema`-based with a relevance `score` (cosine similarity = `1 - distance`) plus document-level citation fields (`document_title`, `document_type`, `source_type`, `source_ref`) so answers are attributable. (4) Retrieval treats chunk content as untrusted data: it returns adversarial text verbatim as evidence and never interprets it; ranking is relevance-only, so injected documents cannot hijack results. Downstream guardrails (the Milestone 9 AI runtime) are responsible for refusing to obey retrieved instructions.
- Consequences: Retrieval is reproducible and network-free in tests via the deterministic embedder; a golden eval corpus + labeled queries and adversarial prompt-injection fixtures guard ranking and injection-resilience, and tenant isolation is proven by unit and live-pgvector tests. Because stale exclusion is a query-time join, retiring a document is instant and reversible via PATCH with no re-embedding. The deterministic embedder is lexical, so real retrieval quality still depends on choosing a production embedding model (ADR-0014 follow-up) wired into both ingestion and retrieval behind the `Embedder` port. Raw embeddings are never returned to API clients or the AI runtime.
- Follow-up: A Temporal `KbIngestionWorkflow`; wire one shared production embedder instance into both ingestion and retrieval; consider a similarity-score floor and a max-context-tokens cap before retrieved chunks enter the AI runtime.

## ADR-0016: V1 AI Runtime Is A Dependency-Free LangGraph-Style Graph Behind Pluggable Ports

- Date: 2026-07-03
- Status: Accepted
- Context: Milestone 9 implements the Python AI runtime (the support agent graph). The default stack names LangGraph for stateful agent orchestration (ADR-0005) and Pydantic-style validation. However, the local engineering harness runs Python through the system interpreter (`pnpm test:py` → `python3 -m unittest`, `pnpm lint` → `python3 -m compileall ai`) on Python 3.14 with no `uv` and no installed third-party packages; LangGraph, LangChain, and Pydantic are therefore not importable, and adding them would make `pnpm test`/`pnpm lint` fail and be non-reproducible for future fresh-context sessions. The runtime's value (state model, node governance, structured outputs, guardrails, traces, evals, escalation) is independent of the specific orchestration library, and the harness explicitly deferred Python dependency management to "when the LangGraph runtime is implemented" (AGENTS.md / TEST_STRATEGY.md).
- Decision: Implement the v1 support graph as a self-contained, dependency-free Python package under `ai/runtime/` that mirrors LangGraph's node model. (1) A tiny graph engine (`graph.py`) reproduces the LangGraph API surface the graph needs — `add_node`/`set_entry_point`/`add_edge`/`add_conditional_edges`/`compile().invoke()` — so nodes are written exactly as they would be for LangGraph. (2) Structured I/O is validated with stdlib dataclasses + explicit validators (Pydantic-equivalent) in `schemas.py`, and `RuntimeResult` mirrors the Temporal `RunAiGraphActivityResult` boundary. (3) The three external dependencies are ports: `ModelProvider` (classification + drafting), `RetrievalPort` (tenant-scoped KB search), and `ToolExecutor` (the Milestone 8 tool-call envelope + permission classes). V1 ships deterministic, offline implementations (`DeterministicSupportModel`, `InMemoryRetrieval`, `InMemoryToolExecutor`) so the graph, evals, and tests run with no network, no wall-clock, and no randomness. (4) Model reasoning goes through the model port; policy and guardrail logic is deterministic Python (safety-critical governance must not be probabilistic). (5) Traces and ids are deterministic (derived from inputs) so runs are reproducible and diffable.
- Consequences: `pnpm test`/`pnpm lint` stay green with the system `python3`, and the runtime is fully unit-/eval-testable offline (49 tests + a golden dataset with hard-fail safety gates). The ports are the exact seams where production adapters plug in: a real LLM-backed `ModelProvider`, a `RetrievalPort` calling `POST /v1/kb/search`, and a `ToolExecutor` calling the live TypeScript tool registry — and, if desired, the real LangGraph library can replace the local engine without changing node code. This partially supersedes the _implementation mechanism_ of ADR-0005 (real LangGraph is deferred, not rejected); ADR-0005's principle (LangGraph-style stateful graph returning structured outputs to Temporal/backend, not owning durable state) is upheld. The deterministic model is rule-based, so real classification/draft quality still depends on wiring a production model behind the port.
- Follow-up: Python dependency management is now provisioned — `uv` is installed (`~/.local/bin`), `ai/.python-version` pins uv-managed CPython 3.12, `ai/uv.lock` is committed, and the harness runs the suite via `uv run --frozen --project ai`; the real AI stack (pydantic, langgraph, langchain-core) is proven to install and is available with `uv sync --project ai --extra llm`. This unblocks (but does not by itself perform) swapping real LangGraph + a provider SDK behind the ports. Remaining: swap the real model/graph adapters; call the live TS tool registry and `/v1/kb/search`; wire the runtime behind the Temporal `RunAiGraphActivity` placeholder; expand the golden dataset to the TEST_STRATEGY §4 counts and add an LLM-graded draft rubric; add versioned prompt files (harness §8).

## ADR-0017: Approval Decisions Persist First, Then Signal; Outbound Sends Reuse The Messages Table With Deterministic Idempotency

- Date: 2026-07-04
- Status: Accepted
- Context: Milestone 10 closes the human approval loop: reviewers resolve pending approvals through the API, the waiting `ticketLifecycleWorkflow` must resume, and approved/edited responses must be delivered to the customer exactly once over email/WhatsApp with a full audit trail. This spans two failure-prone boundaries — the API→Temporal signal (a classic dual-write) and the provider send inside a retryable Temporal activity — and three consumers of the same truth (API, workflow activities, QA/eval).
- Decision: (1) Approval decisions are **persist-first**: one tenant/RLS transaction resolves the approval (guarded by `status = 'pending'` in the UPDATE so concurrent double-decides conflict deterministically instead of overwriting a reviewer) and appends the `approval.{status}` audit event whose metadata carries both `requested_payload` and `approved_payload` — the edited-draft trail QA/eval consumes; only after commit does the API signal `approval_completed` (workflow id `ticket-lifecycle:{tenant}:{conversation}`) through an injectable `ApprovalWorkflowSignaler`. A missing workflow is a reported outcome (`workflow_signal.reason = "workflow_not_found"`), not an error, because seeded/manual approvals have no waiting workflow; transport failures surface as `502 WORKFLOW_ERROR` after the decision is already durable. (2) Outbound messages are rows in the existing `messages` table (`direction: "outbound"`), not a new table: the pre-existing partial unique index on `(tenant_id, idempotency_key)` is the send-once guarantee for the workflow's deterministic `outbound:{tenant}:{ticket}:{approval_id}` key, and the row carries `send_status` (`queued → sent|failed`), `provider_message_id`, `approval_id`, and `ai_run_id` linkage. (3) The production activity implementations (`createApproval`, `sendOutboundMessage`, `recordAuditEvent`) live behind a `TicketLifecyclePersistenceStore` port (database + in-memory) and derive **deterministic ids** — approval id from (tenant, ticket, correlation), audit ids from a stable hash of the write — so Temporal at-least-once retries replay instead of duplicating; a `sent` idempotency key replays the recorded outcome without re-contacting the provider, while a `failed` row is reused by the next retry. (4) Channel delivery goes through pure provider-request builders plus an `OutboundChannelSender` port; the HTTP sender (Mailgun, WhatsApp Cloud) takes an injectable `fetch`, classifies failures as retryable (5xx/transport) vs permanent (4xx/config/credential → `NonRetryableActivityError`), and resolves credentials from the channel config's `send_credential_ref` via an env-backed resolver (no plaintext secrets in rows, §4.1). (5) `approvals.ai_run_id`/`messages.ai_run_id` are linked only when the referenced `ai_runs` row exists (it does not until AI-run persistence lands); the id always remains inside the approval payload, so nothing is lost.
- Consequences: The approval record is the source of truth and every decision is auditable even if signaling fails; the worst dual-write failure mode is a resolved approval whose workflow needs signal redelivery (surfaced by the 502), never a sent-but-unrecorded or double-sent message. Send-once holds across API retries, Temporal activity retries, and concurrent workflows because it is enforced by a database unique index rather than in-process state. The in-memory store and recording sender/signaler keep the whole loop offline-testable; the live workflow suite plus a live-PostgreSQL drive verified pause→resume, send-once, no-send-on-reject, and audit dedup end to end. Trade-offs accepted: audit rows for retried-but-different metadata hash to distinct ids (at-least-once, append-only is fine); outbound email subjects are null until a subject strategy lands (threading relies on RFC 5322 reply headers); `send_status`/`sent_by_type` remain free-text columns pending an enum migration.

## ADR-0018: Observability Is A Shared Port Package; Cross-Process Traces Correlate By Attribute; QA Sampling Is Deterministic

- Date: 2026-07-04
- Status: Accepted
- Context: Milestone 11 adds OpenTelemetry tracing, structured logs with trace ids, metrics for APIs/workflows/AI runs/tool calls/approvals, AI-run persistence with trace links, a QA review surface, and dashboards/alerts. The Compose stack already ships an OTel collector (OTLP 4317/4318) but no Prometheus/Grafana. The repo's testing doctrine requires all of this to be deterministic and offline-testable, and Temporal's workflow sandbox makes strict parent-child span propagation across the API→workflow boundary a nontrivial interceptor integration.
- Decision: (1) A new `@support/observability` package owns the telemetry seam: `startTelemetry` (OTLP/HTTP exporters to the collector, registered global providers, disabled via `OTEL_SDK_DISABLED`), `createInMemoryTelemetry` (test double capturing spans/metrics), a typed `SupportMetrics` port (OTel-backed, no-op, and recording implementations) with canonical instrument names in `SUPPORT_METRIC_NAMES` (`support.{domain}.{measure}`), a JSON structured logger (required fields per DEVELOPMENT_RULES §13, secret-key redaction, active-span trace-id injection), and shared `SUPPORT_ATTR` span attribute keys. Instrumented code takes the port with a no-op default, so telemetry is never required for correctness. (2) V1 tracing correlates across process boundaries **by attribute, not by span parentage**: every span (API `http.request`, worker `activity.*`, `tool.execute`, `approval.decide`) carries `support.correlation_id`/`support.tenant_id`/`support.ticket_id`, `ai_runs.trace_id` joins the persisted run to its spans and to the runtime's redacted trace export, and `audit_events.correlation_id` closes the loop — a ticket is traced end to end by filtering on one correlation id. In-process spans still nest via the active context. Strict W3C context propagation through Temporal (`@temporalio/interceptors-opentelemetry`) is deferred as a follow-up, not rejected. (3) Workflow observability instruments the **activity boundary**: `instrumentTicketLifecycleActivities` wraps every activity with spans/duration/outcome metrics and maps domain outcomes to a closed set of critical failure modes (`ai_graph_failed`, `outbound_send_failed`, `approval_signal_failed`, `event_dead_letter`, `sla_breached`) recorded on one `support.critical_failures` counter that the alert rules key on. Workflow code itself stays deterministic and untouched. (4) `ai_runs` persistence wraps `runAiGraph` (`createPersistedRunAiGraph`): deterministic run ids make Temporal retries replay, failed runs are persisted as evidence with a backfilled deterministic id, and a missing ticket row skips persistence instead of failing the workflow — this is what materializes the Milestone 10 `approvals.ai_run_id`/`messages.ai_run_id` FK links. (5) QA sampling is a deterministic job, not a scheduler: mandatory rules (100% auto-send candidates, 100% high-risk) plus a hash-bucketed random sample (SOP §10 rate), deterministic `qa_review_id` + conflict-safe inserts make re-runs idempotent, and each new review emits `support.qa.review_created.v1`. (6) Dashboards and alert rules are definitions-as-code under `infra/observability/`; the collector re-exposes metrics for Prometheus scraping on `:8889` with `translation_strategy: UnderscoreEscapingWithoutSuffixes` so scraped names match the code's instrument names dot-for-underscore.
- Consequences: Every instrumented path works with telemetry disabled and is asserted offline via the recording/in-memory doubles; the live smoke drive proved spans and metrics arrive at the collector and scrape under the documented names. Flat cross-process traces mean no single waterfall view across API→worker until the Temporal interceptor lands — the trade is zero risk to workflow determinism/replay and no new workflow-bundle dependencies. Metric cardinality stays bounded (route templates, enum attributes). The QA surface (`qa_reviews` CRUD + composite evidence endpoint) gives reviewers conversation, evidence, tool calls, AI output, human edits, and the final response in one tenant-scoped read.
- Follow-up: Adopt `@temporalio/interceptors-opentelemetry` for strict span parentage across the Temporal boundary; ship Prometheus/Grafana (or a hosted backend) in pilot infra and load `infra/observability/`; emit `support.ai_run.completed.v1`/`support.tool_call.completed.v1` domain events from the workflow; wire the workers' structured logger into a production worker entrypoint when it lands (Milestone 10 follow-up); add log export (OTLP logs) once a log backend exists.

## ADR-0019: Security Controls Fail Closed — Deny-By-Default RBAC, Closed Audit Taxonomy, Policy-Version Auto-Send Allowlist, and Safe-Subset Retention

- Date: 2026-07-04
- Status: Accepted
- Context: Milestone 12 hardens the platform for the first pilot: RBAC verification, integration secret handling, PII redaction, a prompt-injection suite, attachment validation, audit completeness, data retention hooks, pilot seed data, the weekly review report, auto-send allowlist controls, and a deployment checklist. The acceptance criteria are absolute ("no known cross-tenant leakage", "no high-risk action can bypass human approval"), so every new control has to default to the safe side and be mechanically checkable, not just documented.
- Decision: (1) **RBAC is deny-by-default with a self-verifying matrix.** The implicit `support_agent` fallback for a missing/empty `x-user-roles` header is removed — no parseable role is `401`, so a misconfigured gateway cannot mint access. `ROLE_PERMISSIONS` is exported as the single source of truth, a new `reports:read` permission gates reporting, and an RBAC matrix test enumerates every registered route via a Fastify `onRoute` collector against a permission catalog: an uncataloged route, a role passing without its permission, or a role denied with it all fail the suite, so adding an endpoint forces an explicit permission decision. (2) **Audit actions are a closed taxonomy.** `SupportAuditActionSchema` in shared-schemas is the canonical action set; the workers audit boundary is typed to it at compile time, the API decide path validates against it at runtime, reserved entries (`policy.*`, `integration.credential_changed`, `permission.*`) pre-commit the families whose write paths are pending, and an audit-completeness test drives every live producer (tool calls stay audited in `tool_calls`, Milestone 8). (3) **Auto-send controls live in the policy model and fail closed.** The tenant allowlist is the active `automation`-domain `policy_versions.content` validated by `AutomationPolicyContentSchema` — an `auto_send_enabled` kill switch plus topics constrained at the contract level to the closed low-risk set (`faq | order_status`), mirroring the AI runtime's hardcoded ceiling so a tenant cannot allowlist `refund` even deliberately (proven by golden case `auto_2`). `GET /v1/policies/automation` resolves the effective controls (missing/inactive/malformed → disabled); `evaluateAutoSendEligibility` in workers is the single gate a future auto-send branch must consult (kill switch → succeeded run → explicit `auto_send` recommendation → low risk → guardrails passed → draft present → allowlisted topic). The v1 workflow still creates an approval unconditionally — a live workflow test drives an `auto_send` recommendation and proves send is unreachable without a human signal. (4) **Retention executes only the reversible-safe subset.** `tenants.retention_policy` (jsonb, migration 0004, `TenantRetentionPolicySchema`) configures per-class windows; the job clears expired `messages.raw_payload_ref` in bounded batches (returning refs for the blob sweeper) and audits `retention.applied`, while attachment/AI-run purges are counted as placeholders until blob-deletion/anonymization strategies exist. No or malformed configuration purges nothing. (5) **Secrets are validated references; PII redaction is content-level.** Both existing resolvers delegate to one `SecretResolver` (`packages/integrations`) that requires `^[A-Z][A-Z0-9_]*$` before touching the environment; the structured logger scrubs emails/phones/card-like digit runs from string content (and the message itself) on top of non-disableable secret-key redaction. (6) **Intake validates attachments before persistence.** A pure `validateInboundAttachments` policy (size cap, content-type allowlist, filename safety, per-message bound) runs at the top of `ingestNormalizedMessage`, so a rejected message creates nothing and is reported per message in the webhook `202` (`rejected`/`rejection_reason`) instead of a retry-provoking error status. (7) **Pilot readiness is executable.** An idempotent, deterministic seed plan (`buildPilotSeedPlan`/`applyPilotSeed`, `pnpm db:seed:pilot`) provisions the pilot tenant end to end — including the global first-party `tool_definitions` rows (closing a Milestone 8 follow-up) and the disabled automation policy — and the weekly pilot report is a tenant-scoped aggregate service behind `GET /v1/reports/pilot-weekly` computing the SOPS §14 metrics in one RLS transaction.
- Consequences: Every Milestone 12 control defaults to deny/disabled/no-purge and has negative tests; the acceptance criteria are backed by suites rather than review (cross-tenant: RLS + RBAC matrix + isolation integration tests; no-bypass: workflow no-bypass test + eligibility gate + Python hard-fail gates incl. the 18-case injection suite; reportability: report endpoint tests; SOPs: SOPS §1.1/§13/§19). Trade-offs accepted: requests that previously relied on the implicit default role now 401 (deliberate breaking change at a placeholder auth boundary — real token verification remains future work at the same choke point); PII scrubbing can redact rare non-PII 13+ digit runs (epoch millis) — an accepted false-positive cost; retention leaves attachment/AI-run data in place until purge strategies land; allowlist changes require an ops-applied policy version until policy write/approve/activate endpoints exist.
- Follow-up: real bearer-token verification and tenant-membership checks in `readAuthContext`; policy write/approve/activate endpoints emitting the reserved `policy.*` audit actions; the auto-send workflow branch behind `evaluateAutoSendEligibility` with shadow mode first (SOPS §17 ladder); blob deletion + anonymization to execute the planned retention classes; scheduling the retention job alongside QA sampling; surfacing `retention_policy` on the tenant API contract.

## ADR-0021: Workflow-Owned Ticket State Persists Through An Explicit Transition Activity; Approval Expiry Is History-Recorded Config; The M13 AI Graph Is A Deterministic TypeScript Stand-In

- Date: 2026-07-04
- Status: Accepted
- Context: Milestone 13 turns the verified workflow pieces into one runnable worker process with real ticket persistence. Three design questions needed answers: (1) how workflow-owned status transitions reach the `tickets`/`ticket_events` rows (BACKEND_SPEC §6.2 requires a ticket event + audit event per transition, and the workflow previously persisted nothing); (2) how approval expiry (BACKEND_SPEC §12) gets a configurable wait inside a deterministic workflow; (3) what implementation backs `runAiGraph` in the production entrypoint, given the Python sidecar bridge is Milestone 14 and the acceptance criterion is "one process runs the entire lifecycle with the deterministic AI model".
- Decision: (1) **Transitions are an explicit `applyTicketStateTransition` activity** the workflow calls at each state change (`waiting_ai` before the AI graph, `waiting_human` at approval request, `waiting_customer` after the send, `closed` on close request; triage and customer-reply moves are folded into `runInitialTriage`/`recordInboundMessage`, and creation into `createOrUpdateTicket`). The store reads the current row status as `from_status` (the row owns truth; the workflow owns sequencing), applies the update plus the `ticket_events` row and the canonical audit event in one RLS transaction, and no-ops when the ticket is already in `to_status` — deterministic `tev_`/`aud_` ids keyed by a per-transition-point `transition_key` make Temporal retries replay instead of duplicate. The store deliberately does not enforce the §6.2 transition matrix: the workflow drives legal sequences, and recording what actually happened beats rejecting recovery paths. (2) **Approval expiry is returned by `createApproval`** (`expires_in_ms`, resolved from `APPROVAL_EXPIRY_MS`, default 24h, non-positive disables) so the value lands in workflow history and replays deterministically; the approval wait races decision signals against the SLA deadline and the expiry deadline as absolute workflow-clock timestamps. `expireApproval` reuses the pending-status guard from the API decide path, so a reviewer decision that wins the race is never overwritten — the workflow then keeps waiting for the signal that is already committed. Expired approvals audit `approval.expired` (a new closed-taxonomy action) and end the workflow in an `approval_expired` phase with the ticket left in `waiting_human`. (3) **`createDeterministicRunAiGraph` is a TypeScript stand-in** mirroring the Python deterministic model's governance (templated commitment-free drafts, always `human_approve`, no tools, medium risk for refunds/cancellations), sharing keyword rules with the new `classifyInitialTriage` triage classifier so triage and drafting agree on topics. It plugs in behind `createPersistedRunAiGraph` unchanged, which is exactly the seam the Milestone 14 sidecar replaces.
- Consequences: Every workflow-owned transition is now queryable (`tickets.status`, `ticket_events`, audit events through the existing read APIs) and idempotent under at-least-once activity execution, at the cost of one extra activity execution per transition (~4 per happy-path workflow — accepted for auditability). Approval expiry changes no default behavior in existing tests (fakes return `expires_in_ms: null`) and is disabled by setting `APPROVAL_EXPIRY_MS=0`. The deterministic TS graph means Milestone 13's live end-to-end drive proves the entire composition — webhook → persisted ticket → drafted approval → API decision → provider send → audit/event trail, across a worker restart — before any network AI dependency exists; its outputs are throwaway by design and never eval-gated.
- Follow-up: Milestone 14 swaps the inner `runAiGraph` for the HTTP sidecar; Milestone 15 replaces both the triage keyword rules and the deterministic drafts with the real model behind the same ports. Multi-run conversations (a customer reply after workflow completion starts a fresh run that re-triages the existing ticket) stay within v1's one-workflow-per-conversation model; refining reopened-ticket semantics is a Milestone 22 item per ADR-0020.

## ADR-0022: The Sidecar Bridge Returns Domain Failures As 200s, The Activity Never Throws For Sidecar Problems, And Service Auth Is A Reserved RBAC Role Behind A Machine Token

- Date: 2026-07-05
- Status: Accepted
- Context: Milestone 14 implements the ADR-0020 HTTP sidecar bridge: the Python runtime as a FastAPI service (`ai/service/`), the production TypeScript `runAiGraph` activity calling it, an internal tool-execution endpoint in the API, and Python HTTP adapters for the `ToolExecutor`/`RetrievalPort` ports. Four contract questions needed answers: how HTTP status codes map onto the runtime's structured-failure model; how the activity reconciles Temporal retry semantics with the acceptance criteria "audited failed AI run, the workflow never fails"; how the sidecar authenticates to the API when `/v1/kb/search` sits behind user RBAC; and how the activity obtains conversation content the workflow input deliberately does not carry.
- Decision: (1) **HTTP status describes the transport, not the run.** `POST /internal/ai/run` returns 200 for both `succeeded` and `failed` `RuntimeResult`s — a failed run is a valid domain outcome (`run_support_graph` never raises, ADR-0016) — while 401 is reserved for auth and 400 for contract violations (malformed/unknown-key bodies, rejected by a strict stdlib parser mirroring the zod schema). The TypeScript activity mirrors this: 400/401/403 and non-contract 200 bodies are permanent failures; transport errors and 5xx are transient. (2) **The activity converts every sidecar problem into a structured `failed` result instead of throwing.** `createPersistedRunAiGraph` persists only returned results, and the workflow must route sidecar outages to human approval rather than fail — so Temporal's activity retry is the wrong lever (exhausting it fails the workflow; throwing skips ai_run persistence). Transient failures are retried in-activity (bounded attempts, backoff, explicit per-attempt timeout) before returning `AI_SIDECAR_UNAVAILABLE`/`AI_SIDECAR_ERROR` with `retryable: true` as recorded evidence; auth/contract problems return permanent codes immediately. Database context/policy loads before the call still throw plain errors — they use Temporal retries like every other persistence activity. (3) **Service-to-service auth is a machine bearer token minting a reserved deny-by-default role.** The API resolves `SUPPORT_INTERNAL_API_TOKEN` (SecretResolver ref conventions) at boot; a constant-time match mints actor `svc:ai-runtime` with the new `internal_service` role, granted exactly `tools:execute_internal` + `kb:search` in `ROLE_PERMISSIONS`. The role is rejected when claimed via `x-user-roles` (a gateway can never mint it), no token configured means the internal surface is unreachable (fail closed), and the sidecar reuses the existing `POST /v1/kb/search` behind its normal permission instead of a parallel internal retrieval endpoint. `POST /internal/tools/execute` wraps the unchanged Milestone 8 `ToolCallRequest`/`ToolCallResult` envelope with `tenant_id`/`ticket_id`/`ai_run_id`/`granted_permissions`; the registry re-enforces permission classes server-side, so the runtime cannot grant itself classes the executor does not honor. (4) **The activity loads conversation context from the database at call time** (`createDatabaseAiGraphContextStore`: messages → runtime roles, customer, tenant brand under RLS) rather than threading message content through workflow history — content stays out of Temporal payloads, and the sidecar receives exactly what the runtime's request schema needs. Tenant automation policy feeds `policy`/`options.allow_auto_send` (the Milestone 12 bridge). (5) **Parity is proven, not assumed:** because runtime ids are content-hashed from inputs (ADR-0016), `service.eval_parity` runs every golden case in-process and through the service path and requires byte-identical results plus passing gates; the deterministic model remains the only provider until Milestone 15.
- Consequences: Sidecar failure modes are observable domain data — `ai_runs` rows with `AI_SIDECAR_*` codes, `ai_graph.failed` audits, human-routed approvals — verified live by the sidecar-down/sidecar-500 e2e drives, and the workflow contract is unchanged (the persistence wrapper and workflow code did not change for the bridge). The reserved-role approach reuses the existing RBAC matrix tests as the security proof surface but leaves real token verification for user auth to Milestone 16 (the machine token is verified; user bearers remain placeholder). In-activity retries mean Temporal's retry policy no longer governs sidecar transport (bounded at three attempts per activity execution); the `routing_decision.priority` vocabulary mismatch (runtime `p1`-`p4` vs platform `p0`-`p3`) is resolved by keeping the workflow-owned ticket priority in the typed contract with the runtime's value visible in `classification`; the runtime's `approval_package` is dropped at the activity boundary until a consumer exists. The sidecar always requires its own bearer token (`SUPPORT_AI_SERVICE_TOKEN` conventions), so a worker configured with a URL but no token fails fast at boot.
- Follow-up: Milestone 15 swaps the model provider behind the unchanged ports and revisits prompt versioning; Milestone 16 replaces placeholder user auth at the same `readAuthContext` choke point; Milestone 18's hardened Compose graduates the sidecar container (non-default tokens, internal-only network); consider surfacing `approval_package` to reviewers once the console consumes it.

## ADR-0023: Real Models Are Config-Selected LangChain Adapters Behind The Unchanged Ports; The Runtime Reports Its Own Provenance; The LangGraph Engine Swap Is Deferred Again

- Date: 2026-07-07
- Status: Accepted
- Context: Milestone 15 implements the ADR-0020 provider-agnostic model layer: the real LLM behind the Python `ModelProvider` port (pilot default Anthropic Claude), the production TypeScript `Embedder` (pilot default OpenAI `text-embedding-3-small`), versioned prompt files, model/prompt/token/cost capture on `ai_runs`, retrieval bounds, and the live eval gate. Open questions needing decisions: whether to swap the ADR-0016 local graph engine for the real LangGraph library; how provider agnosticism is _proven_ rather than claimed; how the worker learns which model actually produced a run (the Milestone 14 provenance constants were composition-time guesses); how embedding-space integrity is enforced without a migration; and how the runtime/platform priority vocabulary mismatch is unified.
- Decision: (1) **The model adapter is one class, selected by env.** `runtime/llm.py` implements `LangChainSupportModel` over `init_chat_model` (`SUPPORT_LLM_PROVIDER`/`SUPPORT_LLM_MODEL`, keys via SecretResolver-style refs with per-provider defaults, per-call timeout + bounded SDK retries + one in-adapter structured-output repair attempt). The deterministic offline model remains the default — real providers activate only by explicit config, and adapter errors raise so `run_support_graph` returns a structured `failed` run that routes to human (harness §16); nothing fabricates output after an error. Sampling parameters are sent only when explicitly configured (current Claude models reject non-default values). (2) **Structured outputs are enforced with closed-vocabulary JSON schemas** (topics, sentiments, urgencies, priorities, sensitive flags as enums) via `with_structured_output`; policy and guardrail logic stays deterministic Python (ADR-0016) — the real model classifies and drafts, it never governs. (3) **Provider agnosticism is proven by a scripted provider, not asserted.** `SUPPORT_LLM_PROVIDER=scripted` runs the deterministic support rules behind the _same_ adapter (prompt files → chat-model interface → structured outputs → usage capture), so the golden + injection suites passing under `scripted` and under a real provider with only env changes is the mechanical proof; `evals.live_runner` is the opt-in live gate that runs both suites against the configured provider with the unchanged hard-fail gates. (4) **The LangGraph engine swap is deferred again (ADR-0016 seam intact).** The local engine reproduces exactly the LangGraph API surface the graph uses; node code is engine-agnostic; a swap would change no observable behavior (byte-identical parity is the proof standard) while adding a runtime dependency to every sidecar deployment mid-launch. Revisit when a real LangGraph capability (checkpointing, interrupts, streaming) is actually needed. (5) **The runtime reports its own provenance and usage.** `RuntimeResult` gains a `model` section (provider, model id, prompt-version map, calls, tokens, latency, cost estimate — aggregated on the run trace); the wire schema carries it optionally; `createPersistedRunAiGraph` prefers it over the static composition-time provenance (which remains the fallback for the TS stand-in and older sidecars) and persists tokens/cost onto `ai_runs`; `ai_runs.prompt_version` stores the sorted prompt ids (they carry their own versions). Prompts are versioned files with stable ids (`runtime/prompts/<prompt_id>.md`, frontmatter-validated); editing a shipped version in place is prohibited — behavior changes mean a new `.v2` file plus an eval run. Cost estimates come from a built-in per-model price table with env overrides. (6) **Embedding-space integrity is metadata, enforced at query time.** Ingestion records `embedding_model_id` on each chunk's jsonb metadata (no migration); chunks without an id are legacy deterministic-embedder chunks by construction; retrieval compares every hit against the active embedder's id and fails closed (`EmbeddingModelMismatchError` → HTTP 409 CONFLICT) instead of ranking across incompatible spaces — the sidecar's retrieval adapter already degrades that to a human-routed failed run. One factory-built embedder instance (`createEmbedderFromEnv`, deterministic default, `openai` allowlisted to 1536-dim-capable models) is shared by ingestion and retrieval in `createDatabaseApiServices`. (7) **Retrieval bounds default to no-op for determinism.** The similarity floor (default 0) and max-context cap (default 24k chars, top hit always kept) are applied in the retrieval service before results reach the AI runtime; defaults deliberately keep the deterministic eval corpus byte-stable, with real floors set per environment (`SUPPORT_KB_MIN_SIMILARITY`; suggested 0.25 for `text-embedding-3-small`). (8) **The runtime speaks the platform priority vocabulary.** `PRIORITIES` becomes `p0`-`p3` (p4 dropped); classifier schemas allow `p1`-`p3` with `p0` reserved for operator-declared incidents; the workflow-owned ticket priority remains authoritative in the typed routing decision (ADR-0022 unchanged).
- Consequences: An LLM provider swap is an env change plus a `live_runner` re-run, and an embedding swap additionally demands a full re-ingest that the system now _enforces_ rather than documents. Every `ai_runs` row records which provider/model/prompts actually produced it with real token/cost numbers, so the pilot's cost telemetry and QA evidence need no joins against deployment configs. All 139 Python and 400+ TypeScript tests stay offline and deterministic; `service.eval_parity` remains byte-identical because the deterministic model's `model` section derives from deterministic fake counts. Trade-offs accepted: the scripted provider proves the plumbing, not real-model quality — the live gates (golden + injection against Claude) are what clear the acceptance criteria, and prompt quality/thresholds remain per-model (SOPS §11); the classifier prompt-injection flagging is now probabilistic where the deterministic model was substring-exact, but governance (human_only forcing, draft suppression, auto-send blocks) stays deterministic Python so a missed detection degrades to the human-approval default, never to auto-send; the retrieval floor default of 0 means real deployments must set their own floor or accept weak evidence reaching (human-reviewed) drafts.
- Follow-up: Run the live gates against the pilot Anthropic default and record pass rates (blocked on API keys, then Milestone 19 re-runs them against live providers); Milestone 18's hardened profile injects the provider keys as real secrets; Milestone 21 expands the golden dataset and adds the LLM-graded draft rubric that scripted-provider runs cannot cover; revisit the deferred LangGraph engine swap when checkpointing/interrupt/streaming capabilities are needed; consider replacing the workers' keyword triage with a classification read from the persisted AI run once multi-run tickets land (Milestone 22).

## ADR-0024: Clerk Is The Pilot IdP; JWT Verification Via jose/JWKS At The readAuthContext Choke Point; DB-Sourced Roles And Membership; Policy Lifecycle Writes With Domain-Exclusive Activation

- Date: 2026-07-07
- Status: Accepted
- Context: Milestone 16 replaces the Milestone 1-15 trusted-header auth placeholder with the ADR-0020 decision (hosted-IdP JWT verification via JWKS) and adds the policy write/activate/archive endpoints so automation-allowlist changes become API-driven. Open questions needing decisions: the concrete IdP vendor (ADR-0020 deferred it to this milestone); the JWT library; how token identity maps onto the existing `users`/`roles`/`user_roles` model; what "tenant membership" means for platform staff; whether the internal machine token moves to IdP-issued client credentials; how policy activation interacts with multiple policies in one domain; and whether the spec's separate version-approve step ships now.
- Decision: (1) **Clerk is the pilot IdP** (dev instance `fluent-leech-80.clerk.accounts.dev`), confirming the ADR-0020 default. The dashboard session-token customization MUST add `{"aud": "support-platform-api", "email": "{{user.primary_email_address}}"}` — default Clerk v2 session tokens carry neither claim, and the audience check is mandatory. The platform depends only on standard OIDC surfaces (issuer, JWKS, RS256 `sub`/`aud`/`exp`), so the IdP is swappable by env change. (2) **Verification uses `jose`** (`createRemoteJWKSet` + `jwtVerify`: cached JWKS with rate-limited refetch on unknown `kid` = restart-free key rotation; issuer/audience/expiry checks with configurable clock tolerance, default 60s; RS256 allowlist; tokens without `exp` rejected) at the same `readAuthContext` choke point every prior milestone hardened — routes, RBAC, and the tool registry see the same `AuthContext` shape as before. Every verification failure maps to the same `401 AUTH_REQUIRED` so responses leak nothing about which check rejected. (3) **Tokens carry identity, the DB carries authority.** The verified `sub` resolves through a `UserDirectory` port (database implementation on the owner connection — auth precedes tenant selection, and platform users have NULL `tenant_id` that RLS would hide) to an active `users` row via the new unique `users.idp_subject` column (migration `0006`); roles come from `user_roles` (global-or-home-tenant grants); an unknown/inactive subject is 401 (no auto-provisioning, no email fallback linking — provisioning sets `idp_subject` explicitly, SOPS §1.1). (4) **Membership is the user's tenant, with NULL meaning platform-wide.** A tenant-bound user selecting any other tenant via `x-tenant-id` is 403 before route handlers run; explicit tenant selection itself remains. (5) **The header mode survives only behind `SUPPORT_AUTH_MODE=insecure-headers`** (default off; JWT mode missing issuer/audience fails the boot rather than silently trusting headers). Unit/e2e suites that exercise business logic opt into it explicitly; production auth behavior is covered by the reworked RBAC matrix running real RSA-signed tokens against a local JWKS server plus an opt-in live Clerk smoke (`RUN_CLERK_LIVE_TESTS=true`) that mints a real session token through the Clerk Backend API. (6) **The internal machine token stays a static shared secret** (Milestone 14 mechanism, checked before user auth in every mode): the sidecar is internal-network-only, no user role holds `tools:execute_internal`, and IdP client-credentials for service-to-service auth would couple the sidecar to the IdP for no pilot benefit — revisit if the sidecar ever leaves the trust boundary (Milestone 18 hardening notes rotation = env change + restart). (7) **Policy lifecycle: create-as-draft, immutable versions, single-shot activation, domain-exclusive active policy.** `POST /v1/policies` creates header + version-1 draft atomically; versions append monotonically and never mutate; activation stamps `activated_at`/`approved_by_user_id` exactly once via a conditional update (re-activation 409), refuses drafts older than the highest activated version, and archives same-domain predecessor policies so the effective-policy read (highest activated version of an active policy) stays unambiguous; archive is manual and fail-closed for automation (safe defaults return). The spec's separate `approve` endpoint is folded into activation for v1 (activator = approver); a two-person rule can split it again with the Milestone 20 console. All transitions emit the reserved `policy.created|activated|archived` audits in the same tenant transaction, and `automation` content is validated against the closed allowlist ceiling at write time AND at activation. (8) **`policies:write` is admin-only** (`platform_admin`, `ops_admin`): the automation domain is a safety control; agents and reviewers keep read access only. `retention_policy` is likewise surfaced read-only on the tenant contract (Milestone 12 follow-up closed); changing it remains an ops action.
- Consequences: No endpoint trusts unverified headers when production auth is on, and the negative suites (absent/expired/forged/wrong-audience/wrong-issuer tokens 401 everywhere; non-member tenant 403; internal endpoints reject user tokens; identity headers ignored under JWT) are enforced by the same self-verifying matrix that already forces a permission decision per route. User provisioning gains one ops step (link `idp_subject`; the pilot seed accepts `PILOT_SEED_*_IDP_SUBJECT`). Automation-allowlist changes are now audited API operations rather than ops SQL, and nothing outside the closed topic ceiling can exist even as a draft. Trade-offs accepted: two DB lookups per authenticated request (user + grants; acceptable at pilot scale — add caching with Milestone 20's rate limiting if needed); the static internal token remains a single shared secret until the trust boundary changes; suspended-user tokens die at the directory lookup, not at the IdP (revocation latency is one request, not a session-store roundtrip).
- Follow-up: Milestone 18's hardened Compose injects `SUPPORT_AUTH_ISSUER`/`SUPPORT_AUTH_AUDIENCE` (production Clerk instance) and rotates the internal token as a real secret; the console (`apps/console`, Milestone 23 per ADR-0026 — originally scoped as a separate repository) shares the same Clerk application and adds token-derived reviewer identity; consider per-request role caching alongside Milestone 20 rate limiting; `integration.credential_changed` and `permission.granted|revoked` remain the last reserved-but-unproduced audit actions (user/role management API is future work).

## ADR-0025: Scheduled Jobs Run As Per-Tenant Temporal Schedules On The Lifecycle Worker; Retention Executes Every Class Fail-Closed Via Sweep-Before-Clear And In-Place Anonymization

- Date: 2026-07-07
- Status: Accepted
- Context: Milestone 17 closes Phase 1's last gap: the QA sampling and retention jobs (Milestones 11/12) were fully tested library functions that nothing invoked — SOPS §10/§16 said "run per tenant on a daily cadence" manually, retention only nulled `messages.raw_payload_ref` and returned the refs "for a storage sweeper" that did not exist, and the attachment/AI-run classes were counted-and-reported placeholders (ADR-0019 follow-up). Decisions needed: the scheduling mechanism, worker topology, blob-deletion semantics against refs whose only production writer is the filesystem raw-payload store (`file://`; no S3/MinIO client exists yet — MinIO is provisioned in Compose but unused), how to purge attachments whose `object_ref`s are mostly provider-side pointers (e.g. `whatsapp-media:` ids) with no locally stored bytes, what "anonymize an AI run" means for `ai_runs` rows whose PII lives inline in `structured_output`, and whether `audit_event_days` executes.
- Decision: (1) **Temporal Schedules, one pair per tenant** (`support-qa-sampling-{tenant}` / `support-retention-{tenant}`), daily at configurable UTC times (defaults 02:00/02:30, 5m jitter), overlap SKIP, 6h catch-up window — Temporal already owns durable execution (ADR-0004), so the jobs inherit retries, visibility, and operator controls (pause/trigger/backfill via UI/CLI) with zero new infrastructure; per-tenant granularity keeps tenants isolated and individually pausable. (2) **The lifecycle worker serves the job workflows on the same task queue** (`qaSamplingJobWorkflow`/`retentionJobWorkflow` exported from the same workflow bundle; activities registered alongside the lifecycle set): one process, one deployment unit, pilot-scale-appropriate — a dedicated jobs queue is a config change away if job load ever competes with ticket traffic. Workflows are thin deterministic drivers looping one bounded-batch activity (500 rows/class, max 50 batches, stop on no-progress) so a big backlog still converges in a single scheduled run without unbounded history. (3) **Bootstrap is create-if-missing from the worker entrypoint and fatal on failure**: every worker start lists active tenants (owner connection) and creates missing schedules; `ScheduleAlreadyRunning` means existing schedules — and operator edits — are never overwritten; a worker that cannot guarantee its schedules exits rather than silently running a system whose jobs never fire. New tenants get schedules on the next worker restart (documented onboarding step). (4) **Blob deletion is a `BlobSweeper` port with sweep-before-clear, fail-closed semantics**: the retention job deletes blobs FIRST and clears/purges only rows whose blobs are confirmed gone (missing file = already-swept success for idempotency); a failed sweep keeps the database ref, is counted, recorded as the new `retention_sweep_failed` critical-failure mode, and retried next run — no blob is ever orphaned by a cleared ref. The v1 implementation is filesystem-only (matching the only production ref writer), refuses paths outside `RAW_PAYLOAD_STORE_DIR` and foreign schemes; an S3/MinIO sweeper lands behind the same port with Milestone 18. (5) **Attachment purge = sweep local blobs, then clear metadata.** Refs are partitioned by scheme: `file://` refs are locally stored bytes and gate the purge fail-closed; anything else (`whatsapp-media:`, URLs) is a provider-side pointer whose bytes we never stored — clearing our metadata IS the purge, and provider-side media retention is governed by the provider relationship, not this job. (6) **AI-run anonymization is in-place column clearing with an explicit marker**: `structured_output` → NULL, `guardrail_results` → `{}`, `anonymized_at` (migration `0007`) stamped; run metadata (status, tokens, latency, cost, provenance, trace id) is deliberately retained so pilot reporting and cost telemetry survive retention. The `anonymized_at IS NULL` filter makes re-runs idempotent and distinguishes "anonymized" from "never had output". (7) **`audit_event_days` stays unenforced**: audit events are the append-only compliance ledger every SOP leans on; purging them is a governance decision deferred until a real tenant requirement arrives (schema keeps the knob). (8) **Job observability is first-class on the metrics port**: `support.job.executions`/`support.job.duration_ms` (job/outcome/tenant) + `support.retention.purged_items` (class/tenant), `job.*` spans, structured per-run logs, and two alert rules (`SupportScheduledJobFailures` on failed runs, `SupportRetentionSweepFailures` on the new critical-failure mode).
- Consequences: Both jobs now run with no human action — the live suite (`pnpm test:jobs`) proves against real local Temporal/PostgreSQL/NATS that bootstrap is idempotent, triggered schedules drive the production worker through both workflows, a retention run clears refs + deletes real blobs from disk + purges attachments + anonymizes runs + audits `retention.applied` with per-class counts, QA sampling queues reviews and emits `support.qa.review_created.v1`, and re-runs are no-ops. Retention's audit metadata shape changed (per-class counts + sweep failures replace the planned-placeholder counts) — deterministic audit ids still dedupe. Trade-offs accepted: schedule coverage lags tenant onboarding by one worker restart (acceptable against running a periodic reconciler; revisit if tenant creation becomes API-frequent); a tenant's daily retention run tolerates up to `50×500` rows per class per day (far above pilot volume; the next run continues the drain); the QA drain loop stops on a zero-sample batch even when the backlog exceeds one batch (unsampled candidates are deterministically never-sampled, so looping further would spin — mandatory candidates are always sampled in their batch); `memory://` refs (in-memory test store) fail the filesystem sweeper by design — unit tests inject the recording sweeper.
- Follow-up: Milestone 18 wires the S3/MinIO object store and its sweeper behind the same `BlobSweeper` port (and the hardened Compose must set `RAW_PAYLOAD_STORE_DIR` consistently for API and worker or move both to S3 refs); Milestone 18's Prometheus/Alertmanager load the two new alert rules; consider a schedule-reconciliation pass (periodic or tenant-creation-triggered) when tenant onboarding stops being restart-adjacent; the WhatsApp media-download slice (Milestone 6 follow-up) will start writing local attachment blobs that this job then actually sweeps; revisit `audit_event_days` enforcement when a tenant contract requires audit purging.

## ADR-0026: The Reviewer Console Lives In This Repository At `apps/console`; The Existing API Is Its Backend; The Console UI Is Milestone 23

- Date: 2026-07-08
- Status: Accepted
- Context: ADR-0020's context stated "the reviewer console will be built in a separate repository, keeping this repo backend-only (ADR-0001)", and `PLAN.md`/`TODO.md`/`PROJECT_HISTORY.md` repeated it; Milestone 20 was scoped as the API surface that separate repository would consume, and the console itself sat in `TODO.md`'s user-owned non-code launch track. Meanwhile the console's contract is weaker than assumed: `packages/api/src/openapi.ts` is a 3277-line hand-written object literal (`buildOpenApiDocument()` imports nothing, so it is not derived from the zod schemas), served at `GET /openapi.json` behind `openapi:read` with no spec file on disk and no test binding it to `routes.ts` or `@support/shared-schemas` — the only assertion checks `openapi === "3.1.0"`. Milestone 20 planned to generate the console's typed client from exactly that spec. Drift is already observable in the docs (`BACKEND_SPEC.md` §17 described approval decisions recording `reviewer_user_id` from the `x-user-id` header, which JWT mode ignores; the code has used `context.actor.userId` since Milestone 16). The user re-opened the decision on 2026-07-08 (`DECISIONS.md` purpose clause: re-litigate when "user direction changes the context") and chose in-repo, no BFF.
- Decision: (1) **The reviewer console lives in this repository at `apps/console`**, added to `pnpm-workspace.yaml` as a new `apps/*` glob. This supersedes ADR-0020's separate-console-repository clause; ADR-0020 decisions (1)-(4) are untouched. ADR-0001 stands as a scope-and-sequencing decision: no console code before Milestone 20 lands the contract. Co-location is not the failure mode `ENGINEERING_HARNESS.md` §"Avoid These Failure Modes" warns about — "building UI before backend contracts" forbids _sequencing_, not _placement_. (2) **The console's contract is `@support/shared-schemas`, consumed through a `packages/api-client` workspace dependency**, not a client generated from `openapi.ts`. Same-language consumers get source-level type sharing; the ADR-0003 precedent of duplicating-and-testing contracts across the TS↔Python boundary does not apply here. Contract drift becomes a `tsc` error rather than a remote runtime discovery. `packages/api-client` stays publishable-shaped so extraction remains cheap, but no registry, version policy, or CI publish token is stood up for a single in-repo consumer. (3) **`openapi.ts` gains a route↔spec drift test at Milestone 20**, reusing the Fastify `onRoute` collector that `rbac-matrix.test.ts` already uses to force a permission decision per route (ADR-0019). This is owed regardless of topology: a hand-maintained spec with no mechanical check is a latent defect, and it is what made the separate-repo plan fragile. (4) **No backend-for-frontend.** `packages/api` is the console's backend: RLS tenant scoping (ADR-0013), deny-by-default RBAC with `ROLE_PERMISSIONS` as the single source of truth (ADR-0019), and Clerk JWT verification at the `readAuthContext` choke point (ADR-0024) — the console shares the same Clerk application, as ADR-0024's follow-up already anticipated. Milestone 20 adds the CORS origin allowlist, the approval evidence composite, queue ergonomics, and Redis rate limiting. A BFF would duplicate the self-verifying RBAC matrix, add a hop, and add a third deploy unit to the ADR-0020(4) single-VM Compose stack; a Next.js app's server-side route handlers already provide a secrets/aggregation seam with no extra deployable. (5) **The console UI is Milestone 23, appended rather than inserted.** Milestones 21 and 22 are referenced by number inside accepted ADRs (ADR-0020, ADR-0021, ADR-0023) and `PROJECT_HISTORY.md`; those are append-only records and renumbering would falsify them. Milestone 23 joins Milestone 20 in Phase 3 — phase membership carries the sequence, the number is an id. Milestone 20 retains only a vertical slice (login → queue → one evidence view → decide), which exists to make its acceptance criterion ("no backend changes required mid-build") falsifiable; that criterion is untestable while no consumer exists.
- Consequences: The console can no longer live in `TODO.md`'s "User-Owned Launch Track (Not Code)"; the remaining user-owned console item is the shared Clerk application. `PLAN.md`'s V2 "human console frontend" entry moves to Milestone 23. Adding a workspace package enrolls it in `pnpm -r lint/typecheck/test/build`, which surfaces two conflicts to resolve at Milestone 20: root `build` is `tsc -p tsconfig.json` per package (a Next.js app needs `next build`), and root `test` chains `pnpm test:py`, so running the default test command requires `uv` + CPython 3.12 — the console loop is `pnpm --filter` scoped. CI is one monolithic job with no path filters (pgvector service + `docker run` NATS + uv), so console changes must be gated behind a `dorny/paths-filter` step _inside_ the existing job; workflow-level `paths:` filters make skipped required checks block merges. There is no frontend test tier today (lint is `tsc --noEmit`, no ESLint, no jsdom, no browser harness), so `TEST_STRATEGY.md` gains one and browser tests stay behind `test:e2e:console`, out of root `pnpm test`. Agent context discipline (ADR-0009) is preserved by a scoped `apps/console/AGENTS.md` whose reading path is the OpenAPI doc and `packages/api-client`, not `BACKEND_SPEC.md`. Trade-offs accepted: the repository grows a second toolchain and a heavier clone; a future frontend-only contributor would see the whole backend. The choice is deliberately the reversible one — extracting a clean `apps/console` later is mechanical, whereas merging two repositories and their histories is not.
- Follow-up: Milestone 20 adds `apps/*` to `pnpm-workspace.yaml`, builds `packages/api-client`, adds the route↔spec drift test, and folds in the `GET /v1/tickets` ergonomics and `ticket_events` read API that Milestone 13's follow-up deferred to "if the console needs them"; Milestone 18's hardened Compose adds a `console` service and extends the TLS reverse proxy beyond the API + webhook endpoints; `SOPS.md` §10's "reviewers work the queue through the API" becomes the documented fallback once Milestone 23 ships, and the SOPS §19 deployment checklist gains console boxes. Re-open this ADR when a dedicated frontend contributor needs repository-scoped access, when a second API consumer ships (the V2 chat widget or mobile) and `packages/api-client` must be published for real, or when console deploy cadence makes shared CI the bottleneck. Data freshness for the queue is polling (`updated_since`/ETag); SSE remains deferred (`TODO.md` Milestone 20).

## ADR-0027: Staging Runs `tsx` Source In Pruned `pnpm deploy` Images On A Single Hardened-Compose Bridge Network; Only Caddy Is Public; Shared Raw-Payload Volume; Health-Gated Deploy With Auto-Rollback

- Date: 2026-07-10
- Status: Accepted
- Context: Milestone 18 turns the locally-runnable stack into a production-shaped staging deployment on one VM (ADR-0020(4)). Several implementation choices were open: how the TypeScript services are containerized given the monorepo consumes its own packages as TypeScript source (every `@support/*` package's `exports` points at `./src/index.ts`, and `worker:start`/`api dev` already run through `tsx` — there is no cross-package compiled-JS path); where the retention blob sweeper's files live now that the API writes raw payloads and the worker's sweeper must delete them (the Milestone 17 follow-up left this as "share `RAW_PAYLOAD_STORE_DIR` or deliver the S3/MinIO sweeper behind the same ports"); the reverse proxy and network posture; how secrets are supplied per the SecretResolver contract; the backup/restore mechanism; and the deploy/rollback flow. Docker on the build host lacks buildx (legacy builder only), but CI (GitHub Actions) has buildx — the Dockerfiles use no BuildKit-only syntax so both work.
- Decision: (1) **Production images run `tsx` against the source, not a compiled `dist`.** A multi-stage build installs the frozen workspace, then `pnpm --filter <pkg> deploy --prod --legacy /app` produces a pruned, isolated `node_modules` for exactly that package's production closure; the runtime stage is `node:24-bookworm-slim`, runs as the non-root `node`/`appuser`, and `CMD ["tsx", "src/…"]`. `tsx` was promoted from a root devDependency to a declared production dependency of `@support/api` and `@support/workers` so `--prod` includes it. Compiling to `dist` was rejected: nothing in the repo imports compiled JS (the package `exports` resolve to `src`), so a `dist` build would also require rewriting every package's export map — invasive, and it buys nothing over `tsx`'s runtime type-stripping (CI already owns typechecking). (2) **The API image is also the migrate/seed runner.** `@support/db` (with its `migrations/` directory) ships inside the API deploy output, so `migrate`/`seed` are profile-gated one-shot Compose services running `tsx node_modules/@support/db/src/{migrate,seed-pilot-run}.ts` on the same image — no separate migrator image. (3) **Raw payloads use a shared named volume**, mounted into both `api` and `worker` at `RAW_PAYLOAD_STORE_DIR=/data/raw-payloads` — the "share the directory" branch of the Milestone 17 follow-up. The S3/MinIO `RawPayloadStore`/`BlobSweeper` implementation is deferred (MinIO stays in the stack for future use and its data is backed up out-of-band); shipping a filesystem volume is a deploy-topology change, whereas an object-store port is new application code with its own tests, out of Milestone 18's scope. (4) **Caddy terminates TLS and is the only publicly-published service.** It proxies only `/v1/*` (which includes the provider webhooks) and `/health`; `/internal/*` is refused at the edge and stays reachable solely on the internal network. Every datastore and app service runs on one `backend` bridge network with no host ports; operator UIs (Grafana, Prometheus, Alertmanager, Temporal UI, MinIO console) bind to `127.0.0.1` for SSH-tunnel-only access. The console is Milestone 23 and does not exist yet, so — contrary to ADR-0026's forward-looking follow-up — Milestone 18's Caddyfile proxies only the API + webhooks; the console route is added when Milestone 23 builds it. (5) **Secrets are single-sourced by blast radius.** `infra/production/.env` (Compose interpolation) holds anything shared across services — datastore passwords and the internal machine tokens that _must_ match across api/worker/ai (ADR-0022), so they cannot drift — while `infra/production/env/<service>.env` holds each service's external provider/IdP keys. Real files are git-ignored; `.example` templates are committed. (6) **Backups are a scheduling-loop container.** `pg-backup` takes a nightly custom-format `pg_dump` to a volume, checksums it, prunes past `BACKUP_RETENTION_DAYS`, and ships offsite via a pluggable `BACKUP_UPLOAD_CMD` hook (the offsite target is user-owned and tool-agnostic); `pg-restore-drill.sh` restores the latest dump into a scratch database and verifies it. (7) **Deploy is health-gated with auto-rollback.** CI builds+pushes the three images to GHCR on a `v*` tag, then (only when the repo variable `DEPLOY_ENABLED=true` and `DEPLOY_SSH_*` secrets are set) runs `deploy.sh` over SSH: it records the current tag, pins the new one, applies additive-only migrations, rolls the app services, and health-gates the API — auto-invoking `rollback.sh` on failure. A `DEPLOY_SKIP_PULL=1` escape hatch supports air-gapped/locally-built images.
- Consequences: Staging is byte-for-byte the same code path as local (`tsx` on source), so "works locally" transfers to staging without a build-artifact translation layer; the cost is `tsx`'s small per-process startup transform and shipping source in the image (~330 MB API, ~540 MB worker, ~430 MB AI — acceptable for a single-VM pilot). The whole stack was brought up and validated locally end to end: all three images build; `docker compose config` is valid; migrations 0001-0007 + the pilot seed apply; the RLS smoke check passes (unset tenant raises, cross-tenant read returns zero rows, cross-tenant insert is rejected by the WITH CHECK policy); the worker connects to Temporal through the real AI sidecar bridge and bootstraps the per-tenant QA-sampling/retention schedules; Caddy serves the API over TLS and 404s `/internal/*`; Prometheus scrapes the collector and loads the nine `Support*` alert rules; Grafana auto-provisions the datasource + `Support Platform Overview` dashboard; Alertmanager routes a synthetic critical alert to the `slack-notifications` receiver (delivery is a no-op until `SLACK_WEBHOOK_URL` is set — the URL is rendered to a file the config reads, never baked into an image); the backup→restore drill passes (checksum OK, 7 migrations + tenant restored); `deploy.sh`→`rollback.sh` flips the running image relA→relB→relA through the health gate; and the live DB integration suite (19 tests) passes against the production-compose database. The single-VM choice keeps a manual-failover recovery story (mitigated by the tested restore). The pinned collector image had to move from an arbitrary `0.121.0` to `0.154.0` because the project's `translation_strategy` exporter key postdates 0.121; pinned image tags are now the version the existing config targets.
- Follow-up: User-owned before staging serves real traffic — provision the VM + DNS A record for `STAGING_DOMAIN`, set the Clerk **production** issuer/audience and the Anthropic/OpenAI keys in `env/*.env`, drop in the Slack webhook, set `DEPLOY_ENABLED=true` with the `DEPLOY_SSH_*` secrets, and run the SOPS §19 checklist on the VM (which is where "serve over TLS on the public domain", "a real synthetic alert reaches Slack", and "live suites against the staging database over the network" are actually closed — locally they were validated on the production-compose stack). Milestone 19 wires the live providers (Mailgun/WhatsApp) and rehearses go-live. The S3/MinIO `BlobSweeper` behind the existing port, a `console` Caddy route + service (Milestone 23), and a cron/systemd alternative to the in-container backup scheduler are deferred. Re-open when the pilot outgrows one VM (managed PostgreSQL / Temporal Cloud are drop-in per ADR-0020(4)).

## ADR-0028: The Reviewer Console Is A Static Vite + React SPA Over `packages/api-client`; Clerk Runs Client-Side Behind An Auth-Provider Seam; A New `GET /v1/me` Closes The Roles Gap Permission-Gated Navigation Needs

The app is enrolled in `pnpm -r build`/`typecheck`/`lint`; `vite build` replaces the Milestone 20 `tsc` build (nothing imports the console's output). (2) **The toolchain reuses what the repo already resolves**: `@vitejs/plugin-react-swc` (the repo already whitelists `@swc/core` builds in `pnpm-workspace.yaml`), `vite@8`/`vitest@4` (already in the tree). The seam yields `{ token, tenantId, identity }`; a `ClerkAuthProvider` is active when `VITE_CLERK_PUBLISHABLE_KEY` is set, and a `TokenAuthProvider` (an operator-supplied bearer token) is the fallback for local dev and the Playwright walkthrough, so the console is drivable end to end without Clerk's network — the API still enforces the real JWT/RBAC (or `insecure-headers` locally). The console mirrors the server's deny-by-default `ROLE_PERMISSIONS` (ADR-0019) purely for _navigation_ — hiding the decide actions from `qa_reviewer`, and the AI-run/QA surfaces from `client_viewer` — while the API remains the real authority (the UI gate is a courtesy, not a control). `/v1/me` joins `packages/api-client`, `API_ROUTES`, and `openapi.ts`, held in sync by the Milestone 20 route↔spec drift test.

The `TokenAuthProvider` fallback means a leaked bearer token is as powerful as in any bearer scheme — acceptable because it is dev/test-only and the production path is Clerk-issued short-lived tokens. - Follow-up: Milestone 23's remaining slices build the queue → evidence → decide loop, the QA surface, the component + Playwright tests, and the `console` service in the hardened Compose + the Caddy static/SPA route (closing ADR-0027's deferred console-route item).
